1. 회원가입 (Sign Up) 테스트 케이스
[TC-1-1] 정상 회원가입

조건: 올바른 이메일 형식, 8자 이상의 비밀번호, 2~20자 사이의 유효한 닉네임(한글, 영문, 숫자, 밑줄) 입력

기대 결과: "확인 메일을 보냈습니다." 안내 메시지가 노출되어야 하며, 아직 자동 로그인(세션 생성)은 되지 않아야 함.

[TC-1-2] 입력 폼 유효성 검사 (클라이언트/서버 로직 검증)

조건: 이메일 형식 미준수, 비밀번호 7자 이하, 닉네임 2자 미만 또는 20자 초과, 닉네임에 허용되지 않은 특수문자(예: @, !, 공백 등) 포함 시도

기대 결과: 제출 전/후로 Zod 검증에 걸려 적절한 에러 메시지가 표시되고 가입이 차단되어야 함.

[TC-1-3] 닉네임 중복 차단

조건: 이미 profiles 테이블에 존재하는 닉네임(대소문자만 다른 경우 포함)으로 가입 시도

기대 결과: "이미 사용 중인 닉네임입니다." 메시지가 표시되며 가입이 차단되어야 함 (사전 체크 기능 검증).

[TC-1-4] 닉네임 동시성 방어 (엣지 케이스)

조건: 동일한 닉네임으로 거의 동시에 여러 건의 가입 요청 발생

기대 결과: 첫 요청만 성공하고, 이후 요청은 DB 트리거(handle_new_user) 및 unique index 제약조건에 의해 롤백되며 "이미 사용 중인 닉네임입니다." 에러가 정상적으로 반환되어야 함.

[TC-1-5] 중복 이메일 가입 방어

조건: 이미 가입 및 인증이 완료된 동일한 이메일로 회원가입 시도

기대 결과: 응답 데이터(identities 빈 배열)를 감지하여 "이미 가입된 이메일입니다. 로그인하거나 '비밀번호를 잊으셨나요?'를 이용하세요."라는 안내가 나와야 함.

[TC-1-6] 미확인 이메일 재가입 시도 (엣지 케이스)

조건: 이전 가입 시 이메일 인증을 완료하지 않은 이메일로 다시 회원가입 시도

기대 결과: 오류 없이 확인 메일이 정상적으로 다시 발송되어야 함.

2. 이메일 인증 (Callback) 테스트 케이스
[TC-2-1] 정상 이메일 인증

조건: 수신된 확인 메일의 링크를 클릭하여 /auth/callback 라우트로 진입

기대 결과: 세션이 성공적으로 수립되고, 서비스의 메인 게시판인 /community (또는 지정된 next 경로)로 리다이렉트 되어야 함.

[TC-2-2] 잘못된 인증 링크 진입

조건: 만료되었거나 임의로 변조된 확인 메일 링크 클릭

기대 결과: 인증에 실패하고 /login?error=auth 페이지로 리다이렉트 되어야 함.

3. 로그인 (Sign In) 테스트 케이스
[TC-3-1] 정상 로그인

조건: 가입 및 이메일 인증이 완료된 이메일과 올바른 비밀번호 입력

기대 결과: 로그인 성공 후 /community 페이지로 리다이렉트 되어야 함.

[TC-3-2] 미인증 계정 로그인 시도

조건: 회원가입은 했으나 이메일 확인을 진행하지 않은 계정으로 로그인 시도

기대 결과: "이메일 확인이 필요합니다." 안내 메시지와 함께 [확인 메일 재발송] 버튼이 화면에 노출되어야 함. 버튼 클릭 시 메일이 재발송되어야 함.

[TC-3-3] 잘못된 자격증명 입력 (보안 검증)

조건: 존재하지 않는 이메일을 입력하거나, 존재하는 이메일이나 비밀번호를 틀리게 입력

기대 결과: 계정 존재 여부를 유추할 수 없도록, 사유를 뭉뚱그려 "이메일 또는 비밀번호가 올바르지 않습니다."라는 공통 에러 메시지만 노출되어야 함.

[TC-3-4] 기 로그인 사용자의 접근 제어 (엣지 케이스)

조건: 이미 로그인하여 세션이 있는 상태에서 직접 브라우저 주소창에 /login 또는 /signup을 입력해 접근 시도

기대 결과: 홈(/) 화면으로 즉시 리다이렉트 되어야 함 (새로고침 문제 해결 검증).

4. 비밀번호 재설정 테스트 케이스
[TC-4-1] 비밀번호 재설정 메일 발송

조건: "비밀번호를 잊으셨나요?" 클릭 후 가입된 이메일 주소 입력 및 전송

기대 결과: 성공적으로 재설정 링크가 포함된 메일이 발송되어야 함.

[TC-4-2] 새로운 비밀번호 설정

조건: 재설정 메일 링크 클릭 -> 콜백 처리 후 /auth/reset 페이지 진입 -> 새로운 비밀번호(8자 이상) 입력

기대 결과: 비밀번호가 정상적으로 변경되고 /community 페이지로 리다이렉트 되어야 함.

[TC-4-3] 새 비밀번호 유효성 검증

조건: /auth/reset 페이지에서 8자 미만의 비밀번호 입력 시도

기대 결과: 제출이 차단되고 오류 메시지가 표시되어야 함.

5. 상태 유지 및 공통 기능 (Header / RLS) 테스트 케이스
[TC-5-1] 로그인 상태의 헤더 UI

조건: 로그인 성공 후 화면 상단 헤더 확인

기대 결과: 사용자의 닉네임과 로그아웃 버튼이 정상적으로 노출되어야 함.

[TC-5-2] 비로그인 상태의 헤더 UI (Graceful degradation 포함)

조건: 로그아웃 상태이거나, 의도적으로 환경변수를 누락시킨 상태에서 헤더 확인

기대 결과: "로그인" 링크만 정상적으로 표시되어야 하며, 전체 페이지 크래시(throw)가 발생하지 않아야 함.

[TC-5-3] 로그아웃 기능

조건: 헤더의 로그아웃 버튼(Server Action) 클릭

기대 결과: 현재 세션이 종료되고 즉시 홈(/)으로 리다이렉트 되어야 함. 이후 권한이 필요한 페이지 접근이 차단되어야 함.
